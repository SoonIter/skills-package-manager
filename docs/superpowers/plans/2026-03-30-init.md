# 20260330-init

## Goal

Build a standalone npm CLI package positioned as a skill package manager similar to pnpm.

Goals for version `0.1.0`:

1. Provide `spm add` / `spm install`
2. Generate and maintain `skills.json`
3. Generate and maintain `skills-lock.yaml`
4. Integrate through pnpm `configDependencies` so skills can be installed automatically during `pnpm install`

## Reference Commands

The core commands should align with pnpm package sources first:

```sh
spm add https://github.com/rstackjs/agent-skills.git#path:/skills/rspack-best-practices
```

The CLI is now unified under `spm`:

```sh
spm add rstackjs/agent-skills --skill rspack-best-practices
```

## Confirmed Design

### Package Structure

The repository uses a monorepo structure:

```text
packages/
  skills-package-manager/
  pnpm-plugin-skills/
```

- `packages/skills-package-manager`
  - npm package name: `skills-package-manager`
  - provides one bin:
    - `spm`: CLI
  - contains the core implementation: manifest / lock / resolver / installer
- `packages/pnpm-plugin-skills`
  - npm package name: `pnpm-plugin-skills`
  - used from `pnpm-workspace.yaml > configDependencies`
  - runs logic equivalent to `spm install` during `pnpm install`
  - may only reuse the core implementation of `skills-package-manager` at source/build time, and must not depend on it at runtime after publishing

### Scope of the Initial Commands

Only these commands are included in the first version:

- `spm add`
- `spm install`

Notes:

- `spm` is the unified CLI entry point, with semantics primarily aligned to pnpm package sources

The first version will not include:

- `remove`
- `list`
- `prune`
- dependency resolution between skills

### pnpm Integration

The integration point uses `configDependencies` in `pnpm-workspace.yaml`, not `package.json`.

Target usage:

```yaml
configDependencies:
  pnpm-plugin-skills: "0.1.0+<integrity>"
```

Then run:

```sh
pnpm install
```

`pnpm-plugin-skills` will automatically read `skills.json` / `skills-lock.yaml` from the project root and complete installation.

### pnpm Plugin Hook Design

Based on pnpm 10 hooks, `pnpm-plugin-skills` should use `preResolution` as the automatic installation entry:

```js
module.exports = {
  hooks: {
    async preResolution(options) {
      // Locate the workspace root from options.lockfileDir
      // Run logic equivalent to `spm install`
    }
  }
}
```

Why choose `preResolution`:

- it can access `lockfileDir`, which makes locating the project root easy
- it runs early enough to ensure skills are synchronized before dependency resolution
- it is clearer than top-level module side effects
- it is a better fit than `updateConfig`

The plugin-side installation logic must be:

- idempotent
- able to no-op quickly
- skipped directly when `skills.json` does not exist
- failed when `skills.json` exists but `skills-lock.yaml` does not

### `skills.json` Structure

Use a structure as close to `package.json` as possible:

```json
{
  "$schema": "https://unpkg.com/spm/schema/skills.schema.json",
  "installDir": ".agents/skills",
  "linkTargets": [
    ".claude/skills"
  ],
  "skills": {
    "rspack-best-practices": "https://github.com/vercel-labs/agent-skills.git#path:/skills/rspack-best-practices"
  }
}
```

Field meanings:

- `installDir`
  - default: `.agents/skills`
- `linkTargets`
  - additional directories where skills should be exposed, synchronized via symlinks
- `skills`
  - similar to `dependencies`
  - key is the skill name
  - value is the specifier

### `skills-lock.yaml` Structure

The lockfile stores replayable resolution results. In the first version it should only record facts required for installation.

Suggested structure:

```yaml
lockfileVersion: '0.1'

installDir: .agents/skills
linkTargets:
  - .claude/skills

skills:
  rspack-best-practices:
    specifier: https://github.com/vercel-labs/agent-skills.git#path:/skills/rspack-best-practices
    resolution:
      type: git
      url: https://github.com/vercel-labs/agent-skills.git
      commit: 6cb0992a176f2ca142e19f64dca8ac12025b035e
      path: skills/rspack-best-practices
    digest: sha256-xxxxx
```

### Specifier Design

The goal is to align with pnpm as much as possible.

Confirmed direction:

- For Git/GitHub/GitLab, use pnpm-style formats whenever possible:

```text
repo.git#ref&path:/relative/path
repo.git#path:/relative/path
```

- file sources should also support a unified `#path:/...` format for parser consistency
- git sources in the manifest should be stored as full `.git` URLs whenever possible, rather than a custom `github:` protocol

Supported input types for the core `add` command should be based on pnpm package sources:

- full git URL
- git subdirectory specifier with `#ref&path:/...`
- local path / `file:` specifier
- npm package form
- other source expressions compatible with pnpm package sources

Core command examples:

```sh
spm add https://github.com/vercel-labs/agent-skills.git#path:/skills/rspack-best-practices
spm add https://github.com/vercel-labs/agent-skills.git#main&path:/skills/web-design-guidelines
spm add file:./my-local-skills#path:/skills/foo
spm add file:./my-local-skills/skills/foo
```

A compatibility entry could later support these inputs and translate them internally into the core specifier format above:

- GitHub shorthand
- full GitHub URL
- GitHub URL pointing to a repo subdirectory
- GitLab URL
- arbitrary git URL
- local path + `--skill`

Example specifiers in the manifest:

```json
{
  "skills": {
    "rspack-best-practices": "https://github.com/vercel-labs/agent-skills.git#path:/skills/rspack-best-practices",
    "web-design-guidelines": "https://github.com/vercel-labs/agent-skills.git#main&path:/skills/web-design-guidelines",
    "foo": "file:./my-local-skills#path:/skills/foo",
    "foo-skill": "npm:@scope/skills-pack@1.2.3#path:/skills/foo-skill"
  }
}
```

### Responsibility of `add`

Core `add` form:

```sh
spm add <pnpm-style-specifier>
```

Responsibilities:

1. parse the user-provided pnpm-style specifier
2. generate a normalized specifier
3. write/update `skills.json`
4. resolve the source and lock the result
5. write/update `skills-lock.yaml`

Boundaries:

- `add` is responsible for declaration + locking
- when the same skill name is added with a different specifier, it should report a conflict by default rather than overwrite silently
- explicit refs provided by the user, such as `#main&path:/...`, should be preserved in the manifest
- `repo --skill <name>` is not part of the core `add`; it is syntax sugar for a later compatibility layer

Typical scenarios:

#### git subdirectory specifier

```sh
spm add https://github.com/vercel-labs/agent-skills.git#path:/skills/rspack-best-practices
```

This can be written as:

```json
{
  "skills": {
    "rspack-best-practices": "https://github.com/vercel-labs/agent-skills.git#path:/skills/rspack-best-practices"
  }
}
```

#### explicit ref + path

```sh
spm add https://github.com/vercel-labs/agent-skills.git#main&path:/skills/web-design-guidelines
```

This can be written as:

```json
{
  "skills": {
    "web-design-guidelines": "https://github.com/vercel-labs/agent-skills.git#main&path:/skills/web-design-guidelines"
  }
}
```

#### file specifier

```sh
spm add file:./my-local-skills#path:/skills/foo
```

This can be written as:

```json
{
  "skills": {
    "foo": "file:./my-local-skills#path:/skills/foo"
  }
}
```

If the user directly passes a single skill directory:

```sh
spm add file:./my-local-skills/skills/foo
```

then it can be written directly as:

```json
{
  "skills": {
    "foo": "file:./my-local-skills/skills/foo"
  }
}
```

### CLI Design

`spm`:

- core CLI
- standard entry point
- command semantics primarily aligned with pnpm package sources

Example compatible usage:

```sh
spm add vercel-labs/agent-skills --skill rspack-best-practices
```

Compatibility-layer responsibilities:

1. parse community-style inputs such as `repo + --skill`
2. convert them to the pnpm-style core specifier
3. call the same internal logic as core `spm add <pnpm-style-specifier>`

For example, it may be translated to:

```sh
spm add https://github.com/vercel-labs/agent-skills.git#path:/skills/rspack-best-practices
```

Notes:

- it is not necessary for compatibility handling to actually spawn a subprocess and call `spm`
- it is better for all entrypoints to share the same JS command handler, with only argument parsing split by entry layer

### Responsibility of `install`

`spm install`:

1. read `skills.json` and `skills-lock.yaml`
2. validate that the lock covers the current declarations
3. materialize installations into `installDir` according to the lock
4. create symlinks for `linkTargets`

Boundaries:

- `install` only consumes the lock and does not resolve sources again
- if the lock is stale or missing, it should fail and instruct the user to run `add` again
- if installation of a single skill fails, the whole command fails

### Consistency Rules Between Manifest and Lock

During `install`, validate:

- every `skills[name]` entry in `skills.json` exists in the lock under the same name
- the lock entry `specifier` exactly matches the manifest
- top-level `installDir` / `linkTargets` in the lock match the manifest

Any mismatch makes the lock invalid.

### Install State and Fast Skip

To avoid doing a full installation every time in plugin mode, write a state file such as:

```text
.agents/skills/.spm-install-state.json
```

Example:

```json
{
  "lockDigest": "sha256-abc",
  "installDir": ".agents/skills",
  "linkTargets": [".claude/skills"],
  "installedAt": "2026-03-30T12:00:00.000Z",
  "installerVersion": "0.1.0"
}
```

On each automatic installation:

1. read `skills-lock.yaml`
2. compute an overall digest
3. compare it with `lockDigest` in the state file
4. skip directly if they match
5. perform the real installation only when they differ

### Installation Directory and Symlink Strategy

Default real installation directory:

```text
.agents/skills/<skillName>
```

For example:

```text
.agents/
  skills/
    rspack-best-practices/
      SKILL.md
```

If `linkTargets` includes `.claude/skills`, use a per-skill symlink strategy:

```text
.claude/
  skills/
    rspack-best-practices -> ../../.agents/skills/rspack-best-practices
```

Rules:

- create one symlink per skill; do not symlink the whole directory as a whole
- only overwrite installation directories/symlinks managed by this tool
- do not overwrite user-created regular files or directories
- a marker file such as `.spm.json` may be written inside each installed skill directory
- version `0.1.0` does not implement automatic prune / delete

### Skill Directory Validation

Before installing a skill, the first version should at least check:

- the target directory exists
- the directory contains `SKILL.md`

Allowed additional resources include:

- `references/`
- `scripts/`
- `assets/`

If the directory exists but `SKILL.md` is missing, an example error is:

```text
Invalid skill at <path>: missing SKILL.md
```

### Cache Strategy

Use a project-local cache directory such as:

```text
node_modules/.cache/spm/
```

It can be used to cache:

- git repo checkout / fetch results
- unpacked npm tarballs
- digest results
- normalized source metadata

### Error Handling and Exit Codes

For `0.1.0`, unify exit codes as:

- success: `0`
- failure: `1`

Typical failure types:

- invalid user input
- source resolution failure
- file write failure
- missing or invalid lock
- installation target conflict
- remote fetch failure

In plugin mode:

- no `skills.json`: skip
- `skills.json` exists but no `skills-lock.yaml`: fail

### Concurrency and Atomicity

A project-level lockfile is recommended:

```text
.agents/.spm.lock
```

Behavior:

- try to acquire the lock before installation
- avoid multiple processes replacing the same directory at the same time
- source fetching may run with limited concurrency
- materialization and symlink replacement should prioritize stability and may run serially

Use a temporary-directory + atomic-replace strategy during installation to avoid half-installed state.

### Test Strategy

For `0.1.0`, cover four layers of testing:

1. unit tests
   - specifier parsing
   - manifest / lock read-write
   - digest calculation
   - path normalization
   - symlink target calculation
2. integration tests
   - `spm add` generates `skills.json`
   - `spm add` updates `skills-lock.yaml`
   - `spm install` installs a local file source
   - `spm install` creates `linkTargets`
3. fixture repository tests
   - git repo with `/skills/foo`
   - local directory with a skill
   - npm package fixture with `/skills/bar`
4. pnpm plugin integration tests
   - run `pnpm install` in a minimal workspace fixture
   - verify automatic plugin loading, real installation, symlink creation, and second-run no-op

### Build and Publish Shape

- `spm`
  - normal CLI package
  - provides `bin`
- `pnpm-plugin-skills`
  - must be a self-contained published artifact
  - must not declare runtime dependencies
  - must not rely on lifecycle scripts
  - `pnpmfile.cjs` should only require the bundled runtime inside the package itself

Suggested published package structure:

```text
pnpmfile.cjs
dist/
  runtime.js
package.json
```

Example `pnpmfile.cjs`:

```js
const { preResolution } = require('./dist/runtime')

module.exports = {
  hooks: {
    preResolution,
  },
}
```

### Recommended Implementation Order for 0.1.0

1. `skills.json` / `skills-lock.yaml` schema and read-write support
2. `file:` source + local skill installation
3. actual install directory + symlink logic for `spm install`
4. local path support for `spm add`
5. git source resolve + install
6. `preResolution` integration for `pnpm-plugin-skills`
7. no-op install state / cache
8. npm source support

### Explicitly Out of Scope for 0.1.0

- automatic prune / delete
- dependency resolution between skills
- registry protocol design
- automatic upgrade / re-lock during install
- complex layered exit codes
- global store design
